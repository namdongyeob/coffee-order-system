# ADR-002 Redisson과 DB 비관적 락

## 상태와 결정일

Accepted. 결정일: 2026-07-09. 튜터 확인은 여전히 권장합니다.

## 맥락과 문제

다수 인스턴스에서 같은 사용자의 주문이 동시에 처리되면 같은 잔액을 읽어 중복 차감할 수 있습니다. [동시성 전략](../architecture/concurrency-strategy.md)의 성공 2건·실패 8건·최종 2,000P 시나리오는 이 위험을 구체화합니다.

## 결정 동인

- 다수 인스턴스의 같은 `userId` 주문 진입을 줄입니다.
- Redis 락의 장애·만료와 무관하게 DB 잔액의 최종 정합성을 보장합니다.
- 잠금의 사용자 체감 대기와 DB 경합을 함께 관찰할 수 있어야 합니다.

## 검토한 선택지

| 선택지 | 장애 동작 | 정합성 영향 | 검증 계획 | 운영 위험 | 판단 |
| --- | --- | --- | --- | --- | --- |
| Redisson + DB 비관적 락 | Redis 락 획득 실패 또는 만료가 있어도 DB 락이 최종 방어선입니다. | `UserPoint` row 잠금으로 음수 잔액을 막습니다. | 같은 사용자 10개 동시 주문에서 성공 2건, 최종 2,000P, 음수 없음 검증을 계획합니다. | Redis 장애 시 DB 경합 증가, 락 TTL·대기 설정 오판 위험. | 채택. |
| DB 비관적 락만 사용 | Redis 장애 영향은 없지만 모든 동시 요청이 DB 경합으로 갑니다. | DB 정합성은 유지합니다. | 동일 동시성 시나리오와 DB 대기·timeout 관찰이 필요합니다. | 다수 인스턴스 고경합에서 DB 부하와 지연 증가. | 보류. 현재는 진입 제어 요구를 충족하지 못합니다. |
| Redisson만 사용 | Redis 장애, 네트워크 단절, TTL 만료 시 보호가 약화될 수 있습니다. | DB 최종 정합성만으로는 보장되지 않습니다. | 락 획득 실패·만료·Redis 비가용 시나리오가 필요합니다. | 락 키·TTL 설정 오류가 잔액 오류로 연결될 수 있습니다. | 제외. 단일 락을 최종 정합성 근거로 둘 수 없습니다. |

## 결정과 이유

Redisson은 주문·결제 진입 제어로 사용하고 DB 비관적 락은 포인트 최종 정합성 보장 장치로 사용합니다. 앞단의 Redisson은 경합을 줄이고, DB 잠금은 Redis 장애나 락 만료에도 잔액을 보호하는 이중 경계입니다.

## 결과와 단점

동시성 실패의 책임을 진입 제어와 최종 저장으로 나누어 설명할 수 있습니다. 반면 락 획득 순서, TTL, timeout, Redis 가용성을 운영해야 하며 두 잠금의 대기 시간이 합쳐질 수 있습니다.

## 검증 현황과 계획

- 실제 근거: 이 ADR 보강 Issue에서는 런타임 동작을 변경하거나 재실행하지 않았습니다.
- 계획된 검증: Level 3에서 동시 주문·최종 잔액을 확인하고, Redisson을 포함하는 구현 Issue에서는 Level 4에서 Redis 비가용·락 획득 실패를 분리 관찰합니다.

## 재검토 조건

- 실제 부하에서 DB 경합 또는 Redisson 대기가 허용 기준을 넘을 때.
- 분산 락의 장애 처리·TTL 정책 변경이 필요할 때.
