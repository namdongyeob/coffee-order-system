# ADR-001 3계층 아키텍처

## 상태와 결정일

Accepted. 결정일: 2026-07-09.

## 맥락과 문제

주문, 포인트, 랭킹, 이벤트 기능이 함께 있는 과제에서 HTTP 처리, 유스케이스, 저장소 접근이 섞이면 동시성 및 인프라 연동의 책임을 검토하기 어렵습니다.

## 결정 동인

- API 흐름과 트랜잭션 경계를 읽기 쉽게 유지합니다.
- 과제의 정합성·동시성 검증 책임을 명확히 합니다.
- 아직 근거 없는 공통 추상화를 피합니다.

## 검토한 선택지

| 선택지 | 판단 |
| --- | --- |
| Controller-Service-Repository | 채택. 현재 기능 규모에서 책임과 흐름을 가장 명확히 드러냅니다. |
| 기능별 Generic Manager 또는 CommonService | 제외. 책임을 숨기고 과제의 검증 대상을 추적하기 어렵습니다. |
| 별도 도메인 계층을 즉시 도입 | 보류. 현재 문서만으로 추가 계층의 문제 해결 근거가 부족합니다. |

## 결정과 이유

기본 아키텍처는 Controller-Service-Repository 3계층으로 구성합니다. 세부 책임과 금지 규칙은 [3계층 설계 정책](../architecture/layered-design-policy.md)을 따릅니다.

## 결과와 단점

요청 흐름과 저장소·인프라 책임을 리뷰하기 쉬워집니다. 반면 복잡한 도메인 규칙이 실제로 증가하면 Service가 비대해질 수 있으므로, 그때는 별도 ADR로 계층 추가 필요성을 판단합니다.

## 재검토 조건

- Service 하나가 여러 도메인 책임을 지속적으로 갖게 될 때.
- 별도 도메인 계층이 중복 제거가 아니라 실제 규칙의 응집도를 높인다는 근거가 생길 때.
